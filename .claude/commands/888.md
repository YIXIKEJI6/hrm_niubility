# Bug 诊断与整改方案

对用户报告的 bug 进行全链路诊断，输出根因分析、整改方案和影响评估。

## 输入

用户提供：截图 + 问题描述（或仅截图）

## 诊断流程

### 第一步：问题识别
从截图和描述中提取：
- 哪个页面/模块出问题
- 具体现象（数据丢失、显示异常、操作失败等）
- 涉及的流程类型（下发/申请/提案/赏金榜/审批等）

### 第二步：全链路追踪
按数据流方向逐层排查，使用 Explore agent 并行调查：

1. **数据库层** — 表结构是否有对应字段、数据是否存在
2. **API 层** — 路由是否返回完整数据、SQL 查询是否遗漏字段
3. **前端数据映射层** — 接收到 API 数据后是否正确映射（重点检查 taskDataMapper、buildSmartTaskData 等工具函数）
4. **组件渲染层** — 组件 props/state 初始化是否正确读取映射后的数据

### 第三步：对比分析
找到同类功能中**正常工作**的流程，对比代码差异：
- 哪些流程用了正确的数据映射？
- 出问题的流程缺少了什么？

### 第四步：关键文件定位
列出涉及的所有文件和具体行号。

## 输出格式

### 诊断结果

```
根因：一句话总结
位置：文件路径:行号
```

### 数据流验证表

| 环节 | 状态 | 说明 |
|------|------|------|
| 数据库 | ✅/❌ | ... |
| API 返回 | ✅/❌ | ... |
| 前端映射 | ✅/❌ | ... |
| 组件渲染 | ✅/❌ | ... |

### 整改方案
- 具体修改内容（含代码片段）
- 修改范围（涉及哪些文件）

### 影响评估表

| 影响范围 | 风险等级 | 说明 |
|----------|----------|------|
| ... | 无风险/低/中/高 | ... |

### 结论
- 是否可以安全修复
- 是否需要用户确认后再执行

## 注意事项

- **先诊断，不要直接改代码** — 必须等用户评估后确认
- 重点关注 RACI/审批/三入口一致性/多流程隔离（参考 feedback_flow_selfcheck 记忆）
- 改动如超出当前节点，必须先提醒用户影响范围（参考 feedback_flow_impact_check 记忆）
- 对比正常流程和异常流程的代码差异是最高效的排查方法
